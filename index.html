Web Platform Specs

the kind, gentle, and fun place to make specs

Web developers create most of the Web’s magic, yet they are almost absent from the process of creation of core Web technology. Web standards suffer sorely from Web developers’ absence; yet, at this time, contributing to standards remains unnecessarily daunting. Web Platform Specs — or “WebSpecs” for short — aims to progressively build up both the technical solution and the community involvement to address this issue.

Read the existing specifications. There aren’t that many yet, but we’re building things up piece by piece. Propose an idea. We have a welcoming forum to which anyone can bring an idea that isn’t yet mature enough to become a specification. Create new Web technology, either by proposing a change to an existing document or by creating a new one from scratch.

What is this?

This project is not a replacement or a competitor but rather a complement to W3C, WHATWG, TC-39, the IETF, or whichever standards organisation may be in charge of this or that bit of the platform. As such, it is emphatically not an xkcd:927 alternative to existing solutions.

Our idea is simple: this project provides a welcoming environment for anyone to contribute to the creation and development of standards, using tools and workflows familiar to Web developers. We take care to ensure that the legal requirements involved in contributing to given standards are fulfilled without getting in the way. Then whatever dark magic is required to get a document produced here published on a standard track within a given organisation does not interfere with the fun work of creating Web standards.

Historically, standards discussions have often been plagued by negativity, distrust, and stop-energy. The standards world is fraught with useless drama and what often looks like petty turf wars. This drives away many would-be contributors since sane people rarely wish to endure abuse for no good reason when all they want is to make the Web a better place.

That stops here.

In this project, we take friendly, constructive interactions very seriously and we take a very dim view of invidious comments. If you notice unfriendly behaviour, please flag it as such as it will not be tolerated. For more details, you can see our anti-jerk policy.

Core Principles

Web standards have flourished in large part through their openness. A core tenet of WebSpecs is to create standards that are fully open, matching the needs and habits of the Web community. This builds on the following principles:

Forkable
Up until a few years ago, forking was perceived as a mode of failure. Today, it is a mode of collaboration: Make Forks Not War. XXX
Hackable
Thoughtful
Standard organisations often start off flexible and smart, but over time collectively suffer from a slow, toxic drift towards dumb adherence to rules over thoughtful consideration of goals. It is a continuing goal of WebSpecs to avoid this drift, to keep the amount of rules to a strict minimum, and to always leave room for intelligent discussion amongst smart, constructive adults.

Software Practices
Tested
There is only one difference between a WebSpecs specification sitting in a topic branch or on anyone’s fork and one that’s taken into the master branch of a webspecs repository: testing. If a specification has a good enough (for sane values of “good enough”, perfection not being of this world) test suite in our sister Web Platform Tests project and that test suite has a good enough pass ratio in at least a couple of major browsers, then that specification is good to merge to master. Otherwise, it’s a proposal.
